## 1. Входная точка: Классификация запросов

### Decision Tree: Куда направить запрос?

```
Получен запрос/задача
│
├─ Внешний источник? (Stepik, студенты, партнёры)
│  └─> requirements/
│      └─> Неоднозначен или требует анализа?
│          ├─ Да → research/
│          └─ Нет → сразу plans/
│
└─ Внутренний источник? (команда, идеи)
   └─> Стратегический (дни/недели)?
       ├─ Да → plans/
       └─ Нет → tasks/
```

### Критерии классификации

**→ requirements/** если:
- ✅ Требование от Stepik платформы
- ✅ Запрос от студентов/преподавателей
- ✅ Требования партнёров/спонсоров
- ✅ Изменения в Scala/инструментах (внешние)

**→ plans/** напрямую если:
- ✅ Внутренняя идея команды
- ✅ Улучшение существующего контента
- ✅ Рефакторинг структуры
- ✅ Новый раздел курса

**→ tasks/** напрямую если:
- ✅ Багфикс
- ✅ Опечатка/мелкое исправление
- ✅ Тривиальная задача (<1 час)

## 2. Сценарии работы

### Сценарий 1: Требование от внешнего источника

**Пример**: Stepik требует поддержку нового формата упражнений

#### Шаг 1: Документировать требование

```bash
# Создать файл
requirements/stakeholders/platform/stepik-new-exercise-format.md
```

```markdown
# Требование: Поддержка нового формата упражнений Stepik

## Источник
- **От кого**: Stepik Platform Team
- **Дата**: 2025-12-30
- **Приоритет**: high
- **Статус**: new

## Описание
Stepik внедрил новый формат интерактивных упражнений
с пошаговой отладкой. Требуется адаптировать упражнения курса.

## Критерии приемки
- [ ] Все упражнения конвертированы в новый формат
- [ ] Тесты работают
- [ ] Документация обновлена

## Влияние на проект
- content/exercises/ (все упражнения)
- Процесс создания упражнений
- Инструменты экспорта
```

#### Шаг 2: Оценить необходимость исследования

**Вопросы:**
- Требование однозначное? → Нет → research
- Несколько вариантов реализации? → Да → research
- Нужен анализ рисков? → Да → research
- Требование простое и понятное? → Нет research → сразу план

**Если нужен research:**

```bash
# Создать задачу исследования
tasks/planning/research-stepik-new-format.md
```

```markdown
## Вопросы для исследования
- Какие варианты миграции существуют?
- Можно ли автоматизировать?
- Какие риски?
- Оценка трудозатрат?

## Результат
research/technical/stepik-new-exercise-format.md
```

```markdown
- **Связанное требование**: requirements/stakeholders/platform/stepik-new-exercise-format.md

## Исследуемые варианты

### Вариант 1: Автоматическая конвертация
- Плюсы: Быстро (2-3 дня)

- Автоматическая конвертация + ручная проверка
- Оценка: 3 дня скрипт + 1 неделя проверка

## Рекомендации
**Выбран вариант 3**: Гибридный подход
- Оптимальный баланс скорости и качества
- Риски минимальны
```

```markdown
# Требование: Добавить больше практических проектов

- **От кого**: Студенты курса (опрос, комментарии)

- **Приоритет**: high
- **Статус**: new

На основе опроса 150 студентов:

- 65% считают проекты важнее теории

Практические проекты:
- Повышают вовлечённость

- Увеличивают completion rate курса

## Критерии приемки
- [ ] 3-5 практических проектов добавлено
- [ ] Проекты покрывают разные области (data, web, CLI)
- [ ] Каждый проект с детальным описанием и решением
```

## 3. Когда проводить Research

### Decision Tree: Нужен ли research?

```
Получили требование/задачу
│
├─ Требование однозначное и понятное?
│  └─ Да → Пропустить research, сразу план
│
├─ Есть несколько вариантов реализации?
│  └─ Да → Research для сравнения
│
├─ Неизвестная технология/подход?
│  └─ Да → Research для изучения
│
├─ Высокие риски?
│  └─ Да → Research для анализа рисков
│
├─ Влияние на всю архитектуру курса?
│  └─ Да → Research для оценки влияния
│
└─ Простая и тривиальная задача?
   └─ Нет research, можно сразу в tasks
```

### Типы research

**1. Технический research** (research/technical/)
```
Цель: Изучить технологию, инструмент, подход
Пример: "Какой формат лучше для упражнений?"
Результат: Рекомендация с обоснованием
```

**2. Педагогический research** (research/pedagogy/)
```
Цель: Найти лучшие практики обучения
Пример: "Как эффективно обучать ФП?"
Результат: Методология и структура
```

**3. Контентный research** (research/topics/)
```
Цель: Изучить тему для курса
Пример: "Что включить в раздел про типы?"
Результат: Структура раздела и топики
```

**4. Конкурентный research** (research/competitors/)
```
Цель: Анализ аналогов
Пример: "Как устроены лучшие курсы Scala?"
Результат: Best practices и идеи
```

### Формат результата research

```markdown
# Исследование: [Название]

## Исследуемые варианты
### Вариант 1: [Название]
- Преимущества
- Недостатки
- Оценка трудозатрат

### Вариант 2: ...

## Выводы
[Что выяснили]

## Рекомендации
**Выбран вариант X** потому что...

## Следующие шаги
- Создать план: plans/...
```

## 4. Работа с планами

### Когда использовать роль planning.md

**Использовать роль** `prompts/modules/roles/planning.md` для:

✅ **Стратегических планов:**
- Roadmap курса
- Разделы с 10+ уроков
- Крупные фичи (практические проекты, видео-уроки)
- Задачи с неопределённостью и рисками
- Планирование с декомпозицией на итерации

❌ **НЕ использовать для тактических:**
- План одного урока
- Простой рефакторинг
- Мелкие улучшения

### Создание плана: Пошагово

#### 1. Подготовка

```bash
# Прочитать связанные документы
requirements/... (если есть)
research/...     (если есть)
plans/course/roadmap.md (контекст)
```

#### 2. Активация роли (для стратегических)

При работе через EnterPlanMode или создании файла в plans/,
Claude автоматически использует роль planning.md согласно CLAUDE.md.

#### 3. Структура плана

```markdown
# План: [Название]

## Метаданные
- Создан: YYYY-MM-DD
- Статус: planned/in-progress/completed
- Приоритет: low/medium/high/critical

## Цель (Outcome-driven)
Что должно быть достигнуто? (результат для студента/проекта)

## Контекст
Почему это нужно? Связь с требованиями/исследованиями.

## Декомпозиция (WBS + Story Mapping)

### Итерация 1: MVP (сроки)
- Задачи
- Критерий завершения

### Итерация 2: Расширение
- Задачи
- Зависимости

## Приоритизация
- P0 (critical): обязательно для MVP
- P1 (high): ключевая ценность
- P2 (medium): улучшает
- P3 (low): можно отложить

## Риски и митигация
```

## 5. Работа с задачами (Kanban)

### Жизненный цикл задачи

```
TODO → IN-PROGRESS → REVIEW → DONE → ARCHIVE
  ↑          ↓
  └── BLOCKED
```

### Ежедневный workflow

#### Утро: Выбор задачи

```bash
# 1. Проверить Kanban
cat tasks/board/todo.md

# 2. Выбрать задачу (сверху вниз по приоритету)
# Критические → Высокий → Средний → Низкий

# 3. Проверить WIP limit (макс 1-3 задачи одновременно)
cat tasks/board/in-progress.md

# 4. Взять задачу
@start-task: TASK-2025-12-30-001
```

Claude Code:
1. Проверяет WIP limit
2. Перемещает из todo.md → in-progress.md
3. Обновляет метаданные задачи (статус, время начала)
4. Открывает файл задачи

#### День: Работа над задачей

```bash
# 1. Прочитать файл задачи
tasks/content/lesson-collections-intro.md
```

## 7. Практические примеры (Walkthrough)

### Пример 1: От требования до результата (полный цикл)

**Исходный запрос**: Stepik требует поддержку Scala 3.4

#### День 1: Документирование

```bash
# 09:00 - Получено письмо от Stepik
# 09:30 - Создать requirements
vim requirements/technical/scala-3.4-support.md
```

```markdown
# Требование: Поддержка Scala 3.4

## Источник
- От кого: Stepik Platform
- Дата: 2025-12-30
- Приоритет: high
- Статус: new

## Критерии
- [ ] Весь код работает на Scala 3.4
- [ ] Обновлены build.sbt
- [ ] Примеры протестированы
```

#### День 1-2: Research

```bash
# 10:00 - Оценить: нужен ли research? → Да (могут быть breaking changes)
# Создать задачу исследования
vim tasks/planning/research-scala-3.4.md

# 11:00-17:00 - Провести исследование
# - Изучить changelog Scala 3.4
# - Проверить breaking changes
# - Оценить влияние на код курса

vim research/technical/scala-3.4-migration.md
```

```markdown
# Исследование: Миграция на Scala 3.4

## Варианты
1. Обновить всё сразу (риск: 2 дня downtime)
2. Постепенная миграция (риск: версии вперемешку)

## Breaking changes
- Изменения в pattern matching
- Новый синтаксис for-comprehensions

## Рекомендации
Вариант 1: быстрое обновление
- Оценка: 3 дня работы
- Риск: средний (breaking changes незначительные)
```

#### День 3: Планирование

```bash
# 09:00 - Создать план (стратегический → использовать роль planning.md)
# Claude автоматически применит роль согласно CLAUDE.md

vim plans/technical/scala-3.4-upgrade.md
```

```markdown
# План: Обновление на Scala 3.4

## Связи
- Требование: requirements/technical/scala-3.4-support.md
- Исследование: research/technical/scala-3.4-migration.md

## Итерации

### Итерация 1: Подготовка (день 1)
- Обновить build.sbt
- Обновить зависимости
- Тестовая компиляция

### Итерация 2: Миграция (день 2)
- Исправить breaking changes
- Обновить примеры кода
- Запустить тесты

### Итерация 3: Валидация (день 3)
- Проверить все уроки
- Обновить документацию
- Загрузить на Stepik
```

#### День 3: Декомпозиция

```bash
# 11:00 - Создать задачи
vim tasks/technical/update-build-sbt.md
vim tasks/technical/fix-breaking-changes.md
vim tasks/technical/test-all-lessons.md
vim tasks/technical/update-docs.md

# Добавить в Kanban
vim tasks/board/todo.md
```

#### День 4-6: Выполнение

```bash

@start-task: TASK-update-build-sbt
# ... работа ...
@complete-task: TASK-update-build-sbt

@start-task: TASK-fix-breaking-changes
# ... работа ...
@complete-task: TASK-fix-breaking-changes

# День 5
@start-task: TASK-test-all-lessons
# ... работа ...
@complete-task: TASK-test-all-lessons

# День 6
@start-task: TASK-update-docs
# ... работа ...
@complete-task: TASK-update-docs
```

#### День 6: Завершение

```bash
# Обновить requirement
vim requirements/technical/scala-3.4-support.md
```

```markdown

## Связи
- Исследование: research/technical/scala-3.4-migration.md (completed)
- План: plans/technical/scala-3.4-upgrade.md (completed)
- Задачи: Все выполнены ✅
```

## 9. FAQ

### В: Когда создавать requirements, а когда сразу plans?

**О**:
- **requirements/** → Внешний источник (Stepik, студенты, партнёры)
- **plans/** → Внутреннее решение команды

### В: Всегда ли нужен research?

**О**: Нет. Research нужен если:
- Неоднозначное требование
- Несколько вариантов реализации
- Высокие риски
- Неизвестная технология

Для простых и понятных задач → сразу план.

### В: Можно ли пропустить plans и сразу создать tasks?

**О**: Да, для:
- Тривиальных задач (<1 час)
- Багфиксов
- Срочных исправлений

Но для всего остального → лучше создать план.

### В: Как понять что задача готова к завершению?

**О**: Проверить критерии выполнения в файле задачи. Все пункты должны быть выполнены.

### В: Что делать если requirements изменилось?

**О**:
1. Обновить requirements/ (добавить в историю)
2. Оценить влияние на research/plans/tasks
3. Обновить затронутые компоненты
4. Создать новые задачи если нужно

### В: Сколько задач держать в in-progress?

**О**: WIP limit = 1-3 задачи. Лучше закончить текущие, чем начинать новые.
