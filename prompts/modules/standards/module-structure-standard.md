# Module: Module Structure Standard

## Purpose

Определяет формальную структуру и семантику модулей в модульной системе промптов.
Служит архитектурным контрактом, описывающим анатомию модуля.

## Specification

### 1. Область применения

Данная структура применяется ко всем типам модулей:

* roles (роли)
* rules (правила)
* standards (стандарты)
* instructions (инструкции)

Все модули системы следуют единой формальной структуре.

### 2. Формальная структура модуля

Каждый модуль состоит из трёх обязательных секций:

```
# Module: <Module Name>

## Purpose
<meta-information>

## Specification
<executable specification>

## Version
<meta-information>
```

**Анатомия модуля:**
* **Заголовок** - идентификатор модуля
* **Purpose** - мета-секция (назначение)
* **Specification** - исполняемая секция (сущность модуля)
* **Version** - мета-секция (версионирование)

### 3. Семантика секций

#### Specification (исполняемая секция)

Является **единственным контейнером сущности модуля**.

**Природа секции:**
* Исполняемая спецификация поведения
* Интерпретируется LLM как инструкции к действию
* Определяет поведение при включении модуля

**Допустимое содержимое:**
* Поведенческие инструкции
* Логические правила
* Методология работы
* Ограничения и инварианты
* Допустимые и недопустимые действия

#### Purpose (мета-секция)

**Природа секции:**
* Описывает **зачем** существует модуль
* Контекст для человека-разработчика
* Не интерпретируется как инструкции

**Характеристики:**
* Не содержит поведенческих инструкций
* Не влияет на исполнение модуля
* Может быть удалена без изменения поведения

#### Version (мета-секция)

**Природа секции:**
* Метаданные версионирования
* История изменений модуля

**Допустимое содержимое:**
* Номер версии
* Дата создания/обновления
* История изменений (changelog)
* Статус модуля

**Характеристики:**
* Не влияет на исполнение модуля
* Может быть удалена без изменения поведения

### 4. Именование секций

**Обязательные названия секций:**
* `Purpose` - для мета-описания назначения
* `Specification` - для исполняемой спецификации
* `Version` - для версионирования

Названия секций являются частью формального контракта.

### 5. Принцип разделения исполняемого и мета-контента

**Архитектурный инвариант:**

Модуль разделён на две категории контента:

1. **Исполняемый контент** (Specification)
   - Интерпретируется LLM
   - Влияет на поведение
   - Является сущностью модуля

2. **Мета-контент** (Purpose, Version)
   - Не интерпретируется как инструкции
   - Не влияет на поведение
   - Служит для человека-разработчика

Смешивание категорий нарушает архитектуру модуля.
