# Module: Module Creation Rules

## Purpose

Определяет правила, ограничения и требования к содержимому модулей.
Дополняет module-structure-standard.md нормативными требованиями к качеству и валидации.

## Specification

### 1. Запреты на содержимое в Specification

В секции `Specification` **ЗАПРЕЩЕНО** размещать:

* версии, даты, историю изменений
* пояснения «для читателя»
* комментарии о процессе разработки
* мета-описания назначения модуля
* любую информацию, не влияющую на поведение

**Правило:** Specification содержит только исполняемую логику.

### 2. Требования к наименованию

**Обязательное название для контейнера сущности:** `Specification`

**Запрещённые альтернативы:**
* `Содержание`
* `Описание`
* `Информация`
* `Основное`
* любые производные

**Причина:** Название должно отражать исполняемую природу секции.

### 3. Инвариант модульной системы

**Критическое требование:**

> Удаление секций `Purpose` и `Version` НЕ ДОЛЖНО изменять поведение модуля.

Если удаление мета-секций меняет поведение - модуль некорректен.

### 4. Критерии валидации модуля

Модуль считается **корректным**, если:

* ✅ Вся исполняемая логика в `Specification`
* ✅ Мета-информация полностью отделена
* ✅ Структура соответствует стандарту
* ✅ Секции не дублируют друг друга
* ✅ Модуль интерпретируется без внешних пояснений
* ✅ Инвариант мета-секций выполняется

Нарушение любого критерия - ошибка модуля.

### 5. Принципы создания новых модулей

**Правило минимализма:**
* Новый модуль создаётся только при **доказанной необходимости**
* Три повторения - минимум для выделения модуля (DRY)
* Предпочитай плоскую структуру, пока сложность не обоснована

**Правило переиспользования:**
* Сначала ищи существующие модули
* Композиция предпочтительнее создания нового
* Ясность важнее абстрактной универсальности

**Правило самодостаточности:**
* Модуль не требует устных пояснений
* Модуль работает при переформулировке задачи
* Модуль даёт стабильный результат при повторных запусках

### 6. Приоритет нормативных модулей

**Иерархия приоритетов:**

1. `module-structure-standard.md` - высший приоритет
2. `module-creation-rules.md` - этот модуль
3. Остальные модули системы

При конфликте требований применяется модуль с более высоким приоритетом.

### 7. Ограничения на типы модулей

**Обязательные для всех типов:**
* roles (роли)
* rules (правила)
* standards (стандарты)
* instructions (инструкции)

**Правило:** Исключений из стандарта нет. Все модули следуют единой структуре.
